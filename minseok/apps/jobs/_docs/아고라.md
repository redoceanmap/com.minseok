## 붙여 넣기용 프롬프트

다음 요청을 구현한다. **코드·스키마·API를 쓰기 전에** (1) 성공 기준을 한 문장으로 재진술하고, (2) 아래 `[가정]`·`[미정]` 중 답이 없으면 **질문 목록만** 먼저 출력하고 구현은 중단한다. (3) 범위 밖 아이디어는 “나중에”가 아니라 **별도 섹션 `선택 확장(이번 스프린트 제외)`** 한 줄씩만 적는다.

### 배경

- 참여자(또는 심사자)가 **등록된 작품**에 대해 투표·평가한다.
- 동시 사용자 수·배포 환경·기술 스택은 `[가정]`에 적힌 것만 따른다. 비어 있으면 질문한다.

### 성공 기준 (검증 가능)

다음을 **모두** 만족하면 완료다.

1. **작품 단위**로 평가 대상이 식별되고, 목록(또는 단건) 조회가 가능하다.
2. **투표 규칙**이 문서·코드·테스트 중 최소 한 곳에 명시되어 있고, 그 규칙대로 동작한다. (예: 1인 1작품 1표, 점수제, 기한 내만 허용 등 — 구체 규칙은 `[미정]`이면 질문 후 확정.)
3. **중복·부정 투표 방지**가 “우리가 채택한 신원 모델”에 맞게 구현된다. 신원 모델이 `[미정]`이면 질문한다.
4. **집계** 결과를 조회할 수 있다. (합계·순위·평균 등은 채택한 규칙에 맞게 최소 한 형태.)
5. 자동화된 **검증**: 규칙을 깨는 입력에 대한 테스트 또는 재현 가능한 수동 절차가 있다.

### 범위 (이번에 만든다)

- 작품 CRUD 중 **평가에 필요한 최소 필드**만. (제목·식별자·(선택) 설명 수준; 장식 필드 추가 금지.)
- 투표/평가 제출, 집계 조회, 규칙 검증.

### 범위 밖 (이번에 만들지 않는다)

- 실시간 채팅, 추천 알고리즘, 복잡한 관리자 UI, 다국어, 결제, 범용 권한 RBAC 전체.
- “나중에 쓸 수도” 같은 이유만으로 추상화·플러그인 구조를 추가하지 않는다.

### 구현 순서 (각 단계 끝에 검증)

1. **데이터 모델·규칙 문장 확정** → 검증: 한 페이지짜리 `RULES.md` 또는 동등한 주석으로 규칙을 읽으면 모호함이 없어야 한다.
2. **저장·조회 API(또는 함수)** → 검증: 작품 N개·유저 M명에 대해 규칙 위반 시 거절된다.
3. **집계** → 검증: 알려진 fixture 입력에 대해 기대 합계/순위와 일치한다.
4. **에지 케이스** → 검증: 마감 후 제출, 존재하지 않는 작품, 동시 요청 등 **실제로 일어날 수 있는** 것만 다룬다. 불가능한 경로를 위한 방어 코드는 쌓지 않는다.

### 코드 품질 (하네스)

- 요청과 무관한 리팩터·포맷 정리·데드 코드 삭제를 하지 않는다.
- 더 단순한 해법이 있으면 **구현 전** 한 단락으로 제안하고, 채택 여부를 따른다.
- diff의 모든 변경은 위 성공 기준·범위와 직접 연결되어야 한다.

### 채우기 칸 (요청자가 먼저 적는다)

```text
[가정] 기술 스택 / 배포:
[가정] 대략 동시 사용자 규모:
[미정] 투표 규칙 (1인 1표 vs 점수 vs 복수 표 등):
[미정] 신원 (로그인 / 익명+디바이스 / 초대 링크 등):
[미정] 평가 기간·마감 시각이 있는지:
```

## 이 문서를 쓸 때의 메모

- 프롬프트가 길수록 모델이 범위를 넓히려 한다. 위 **범위 밖** 목록을 지키게 한다.
- “멋진 투표 UX” 같은 모호 목표는 **성공 기준 1~5**로만 받는다.
- 구현 후 요구가 바뀌면 **새 프롬프트**로 규칙을 고치고, 기존 성공 기준과의 차이를 한 줄로 명시한다.
